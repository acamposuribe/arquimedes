# Practice Domain Phase 1: Prompting, Reflection, and Figure Weighting - Implementation Plan

> **Status:** Implemented in code/tests
> **Date:** 2026-04-27
> **Companion spec:** None yet. For this scoped rollout, this plan is the working source of truth until a broader domain-behavior spec is needed.
> **References:** [docs/developer/PLAN.md](../../PLAN.md), [docs/developer/PIPELINE.md](../../PIPELINE.md), [Phase 3 enrichment design](../completed/specs/2026-04-04-phase3-enrichment-design.md), [Phase 6 lint design](../completed/specs/2026-04-05-phase6-lint-design.md), [Collection graph architecture](../specs/2026-04-09-collection-graph-design.md)

## Goal

Introduce a domain-aware Phase 1 behavior layer so `practice` no longer inherits the same LLM voice, emphasis, and reflective framing as `research`.

This rollout focuses on four user-facing outcomes:

1. Practice outputs are written in Spanish.
2. Practice prompts are oriented toward practical use rather than knowledge expansion.
3. Practice reflection prompts ask for practice-focused synthesis rather than research-focused synthesis.
4. Practice reflections and enrichment give more weight to figures.

## Non-goals

- No database schema migration.
- No renaming of internal stored keys such as `methodological_conclusions`, `main_content_learnings`, `why_this_concept_matters`, or `why_this_collection_matters`.
- No change to cross-domain bridge isolation rules.
- No rewrite of existing stored Practice artifacts without rerunning enrichment/lint.
- No attempt in this slice to redesign graph-maintenance findings, which remain corpus-level and shared.

## Current implementation snapshot

Today the pipeline already tracks `domain`, but the LLM behavior is still mostly shared:

- document enrichment prompt text is hardcoded and research-oriented in `src/arquimedes/enrich_prompts.py`
- chunk enrichment explicitly forces English
- figure enrichment is shared across domains and does not distinguish practical vs research reading priorities
- local clustering, concept reflection, collection reflection, and global bridge prompts all speak in a research-librarian voice
- reflection evidence budgets use shared fixed figure caps
- compiled wiki section labels are shared and English-first

This means Practice already has separate storage paths and page trees, but not a separate prompting personality.

## Phase 1 design decision

Keep the internal artifact schema stable and change behavior through a new domain-profile layer.

That means:

- the pipeline still stores the same keys and JSON shapes
- Practice uses different prompt instructions, output language, reflection framing, figure weighting, and user-facing section labels where generated by code
- Research keeps its current behavior unless a later follow-up explicitly changes it

This is the lowest-risk way to make Practice meaningfully different without forcing index, memory, search, or migration work.

## Prompt organization rule

All new Practice prompt text for this phase should live in one dedicated practice-oriented file so it can be reviewed in one place.

Preferred shape:

- `src/arquimedes/practice_prompts.py` - all Practice prompt text and prompt-fragment helpers only

Rules:

- keep only prompt text and prompt-text helper functions in this file
- do not move generic pipeline logic, parsing, or routing into it
- existing modules such as `enrich_prompts.py`, `cluster.py`, and lint reflection modules should call into this file when `domain == "practice"`
- Research prompt text can remain where it already lives for Phase 1

This adds a small amount of indirection, but it should not create serious friction if we keep the boundary clean: one file for Practice prompt content, existing files for orchestration and schemas.

## Success criteria

- Practice document, chunk, figure, clustering, concept-reflection, collection-reflection, and Practice-domain global-bridge prompts are domain-aware.
- Practice prompt outputs are requested in Spanish.
- Research prompt behavior remains backward-compatible.
- Practice reflection evidence includes more figures and prompt instructions treat figures as primary evidence when present.
- Generated Practice-facing page labels that come from code can be rendered in Spanish without changing stored field names.
- Tests cover prompt routing, language/framing differences, and domain-aware figure weighting.

## Implementation strategy

### Step 1: Add a domain-profile layer

Create one small module that centralizes domain behavior instead of scattering `if domain == "practice"` checks across prompt strings.

Proposed module:

- `src/arquimedes/domain_profiles.py` - new
- `src/arquimedes/practice_prompts.py` - new

Responsibilities:

- define the built-in `research` and `practice` profiles
- expose helpers such as `get_domain_profile(domain)` and `domain_output_language(domain)`
- hold prompt-facing values only:
  - output language
  - narrator/persona text
  - document intent
  - reflection intent
  - figure priority
  - user-facing generated labels for compiled sections

Phase 1 should keep this lightweight and code-defined. Config overrides can come later if the profile surface proves stable.

### Step 2: Make enrichment prompts domain-aware

Files:

- `src/arquimedes/practice_prompts.py` - new
- `src/arquimedes/enrich_prompts.py` - modify
- `src/arquimedes/enrich_document.py` - modify only if prompt-version stamping needs to reflect domain-aware prompting
- `tests/test_enrich_prompts.py` - extend
- `tests/test_enrich_document.py` - extend if prompt-version or output expectations change

Changes:

- document prompt:
  - Research stays as-is in spirit.
  - Practice prompt text is defined in `practice_prompts.py` and asks for practical synthesis, operational relevance, reusable rules, actionable cautions, and direct language in Spanish.
  - Keep JSON keys unchanged.
- chunk prompt:
  - remove the current hardcoded English requirement
  - Practice summaries and keywords should be requested in Spanish from `practice_prompts.py`
  - Practice chunk summaries should preserve practical constraints, procedural guidance, requirements, and decisions when present
- figure prompt:
  - Practice instructions from `practice_prompts.py` should emphasize figures as operational evidence, not just descriptive support
  - ask for Spanish descriptions/captions when the material is in Practice
  - preserve current deletion/artifact rules for non-substantive scans
- metadata-fix prompt can remain language-neutral unless tests show a strong need to specialize it

### Step 3: Make clustering and reflective lint domain-aware

Files:

- `src/arquimedes/practice_prompts.py` - extend
- `src/arquimedes/cluster.py` - modify
- `src/arquimedes/lint_collection_reflection.py` - modify
- `src/arquimedes/lint_concept_reflection.py` - modify
- `src/arquimedes/lint_global_bridge.py` - modify
- `tests/test_lint.py` - extend
- clustering tests - extend where prompt text is asserted or mocked

Changes:

- local cluster prompt:
  - Research can keep a broad analytical umbrella style.
  - Practice prompt text should live in `practice_prompts.py` and prefer names and descriptors that help organize practical knowledge, working methods, recurring constraints, and reusable solutions.
- concept reflection prompt:
  - Practice prompt text should synthesize "what this concept is useful for in practice", key takeaways, tradeoffs, missing operational information, and next sources to make the concept more usable
  - keep the same output fields
- collection reflection prompt:
  - Practice prompt text should describe what the collection helps someone do, compare, check, or apply
  - `main_tensions` should lean toward operational tensions, constraints, tradeoffs, and ambiguities
  - `open_questions` should ask what remains unclear for practical use, not generic research questions
  - `helpful_new_sources` should prefer codes, precedents, standards, technical notes, implementation examples, and directly comparable cases when relevant
- global bridge prompt:
  - only the Practice-domain bridge pass changes voice and framing
  - Practice bridges should synthesize reusable patterns across collections, not only analytical through-lines

### Step 4: Increase Practice figure weighting

Files:

- `src/arquimedes/lint.py` - modify
- `src/arquimedes/lint_collection_reflection.py` - modify
- `src/arquimedes/lint_concept_reflection.py` - modify
- `tests/test_lint.py` - extend

Changes:

- replace shared fixed reflection figure caps with domain-aware caps
- proposed starting point:
  - Research: keep current values
  - Practice: raise figure budget for concept and collection reflection passes
- update reflection prompts so Practice treats figures as primary evidence when they contain concrete practical knowledge
- keep chunks available as support, but stop implying that chunks are always the stronger evidence path for Practice

This step changes emphasis, not schema.

### Step 5: Practice-facing generated labels in compiled pages

Files:

- `src/arquimedes/compile_pages.py` - modify
- compile-related tests - extend

Changes:

- where headings are generated by code, render Spanish labels for Practice pages
- keep file paths and internal storage keys unchanged
- initial target labels:
  - `Summary`
  - `Material Conclusions`
  - `Methodological conclusions`
  - `Main content learnings`
  - `Key Concepts`
  - `Figures`

This should be done only where the page's domain is known. Research remains unchanged.

### Step 6: Minimal config and prompt-version hygiene

Files:

- `config/maintainer/config.example.yaml` - modify if we expose new prompt-version guidance or comments
- `docs/developer/PLAN.md` - keep linked and current

Changes:

- if domain-aware prompting changes the semantic contract enough, bump the relevant prompt version string(s)
- keep the versioning simple: avoid a large config surface in Phase 1

## Dependency order

```text

    |
    +--> practice_prompts.py
    |
    +--> enrich_prompts.py
    +--> cluster.py
    +--> lint_collection_reflection.py
    +--> lint_concept_reflection.py
    +--> lint_global_bridge.py
    +--> compile_pages.py

    |
    +--> concept / collection reflection behavior
```

Land the domain-profile layer and `practice_prompts.py` first. Everything else should consume them rather than re-declaring the same rules.

## Testing plan

Before calling the work done:

- run prompt-builder unit tests
- run lint/reflection prompt tests
- run any compile-page tests affected by label changes
- run focused tests for Practice-domain figure weighting

Expected commands, to be refined at implementation time:

```bash
pytest tests/test_enrich_prompts.py
pytest tests/test_enrich_document.py
pytest tests/test_lint.py
pytest tests/test_compile.py
```

If the slice grows much beyond prompt plumbing and tests start taking longer to iterate on, suggest running `/cost` to check cache ratio.

## Rollout notes

- Existing Practice artifacts will not change until the relevant enrich/lint commands are rerun.
- Research artifacts should remain stable if tests confirm prompt routing is domain-aware and default-safe.
- If this Phase 1 slice lands cleanly, a later Phase 2 can revisit stored artifact semantics and section names without mixing migration work into the prompt split.

## Resolved decisions

1. Practice canonical concept names and bridge names are generated in Spanish.
2. "Everything in Spanish" includes generated wiki section headings for Practice pages.
3. Practice `helpful_new_sources` explicitly biases toward standards, regulations, manufacturer documentation, and built precedents when relevant.

## Definition of done

- New plan-linked implementation lands without schema churn.
- Practice prompt behavior is visibly different from Research in tests and code.
- Practice outputs are requested in Spanish.
- Practice reflection passes give more importance to figures.
- `docs/developer/PLAN.md` and this plan stay aligned with the implemented scope.
